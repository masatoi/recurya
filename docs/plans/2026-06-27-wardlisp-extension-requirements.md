# wardlisp 拡張要件（ノベルゲーム基盤のための前提）

> 作成日: 2026-06-27
> 動機: [ノベルゲーム基盤設計](./2026-06-27-novel-engine-design.md) が wardlisp を「シーンを評価して演出データを返す」用途で使うために必要な言語拡張の**要件**を定める。
> 位置づけ: **要件のみ**。設計・実装は **別セッション**で `masatoi/wardlisp` リポジトリに対して行う（現状チェックアウト: `/home/wiz/recurya/.qlot/local-projects/wardlisp/`、ブランチ `feat/random-builtin`、v0.2.0）。

## このドキュメントの目的

ノベルエンジンは「`===scene===` の wardlisp 式を、読者フラグを注入して評価し、返り値（演出ディレクティブのデータ）をホストが走査して描画する」方式（設計 §確定判断 1〜5）。これを成立させるために wardlisp に必要な機能を、ノベル側の利用シナリオから逆算して列挙する。**何を作るか（要件・受け入れ条件）**までを定め、**どう作るか（実装設計）**は次セッションに委ねる。

## wardlisp 現状（実機調査 2026-06-26）

- 最小 Lisp。値: 整数(CL integer)、浮動小数(double)、`t`/`nil`、**シンボル(=小文字化した CL 文字列)**、リスト(実行時は `ocons` struct)、closure、builtin。
- **文字列リテラル/型が無い**。`"hello"` は `Unexpected character: "` で reader が拒否（`reader.lisp` の `read-expr`/`atom-char-p` が `"` を非対応）。文字列操作 builtin も無い。
- 日本語テキストはクオートシンボルとして部分的に持てる（`'おはよう、…`）が、ASCII 空白/記号・改行・エスケープ不可で本文には脆い（採用しない）。
- サンドボックス: `evaluate code &key fuel max-depth max-cons max-output max-integer timeout random-seed` → `(values 結果 metrics)`。メモリは `track-cons`/`exec-ctx-max-cons` で会計。
- パッケージ: `:wardlisp` は **`evaluate` と `print-value`**（＋条件型）のみ公開。`wardlisp/src/types` は `ocons/ocons-p/ocons-ocar/ocons-ocdr/make-ocons` 等を公開。
- ソース: `src/{types,reader,evaluator,builtins,env,main}.lisp`。

## 設計上の要点：文字列とシンボルの衝突

現状「シンボル＝裸の CL 文字列」のため、文字列値を単純に CL 文字列で表すと**シンボルと区別できない**。したがって文字列は**専用の型でボックス化**し、シンボルは従来通り裸の文字列のまま、という分離が要る（実装方針の核。詳細設計は次セッション）。

## 要件

### R1. 文字列型（必須）

ノベルの会話本文・地の文を保持・評価・印字できること。

- **R1.1 リテラル構文**: reader が `"..."` を文字列として読む。最低限のエスケープ `\"` `\\` `\n` `\t` をサポート。`atom-char-p` が既に `"` を除外しているので、`read-expr` に文字列分岐を追加する形になる（実装注記）。
- **R1.2 値型**: 文字列はシンボル（裸の CL 文字列）と**区別可能な専用型**（ボックス化）。
- **R1.3 自己評価**: 文字列リテラルは自身に評価される（シンボル探索の対象にならない）。
- **R1.4 印字**: `print-value` は文字列を `"..."`（エスケープ復元）で印字し、シンボルは従来通り裸で印字する。両者が見分けられること。
- **R1.5 等価**: `equal?`（および該当する比較 builtin）が文字列を**内容比較**できる。
- **R1.6 メモリ会計（セキュリティ）**: 文字列の生成（リテラル評価・後述の文字列操作）が**文字長に比例して** `max-cons`（または等価なメモリ予算）を消費し、巨大文字列・ループ生成を既存のサンドボックス境界で抑止できること。リテラル長にも上限を設ける（整数リテラルの 100 桁上限と同様の方針）。

**受け入れ条件（ノベル側の利用）**:
- `(evaluate "\"おはよう、元気？\"")` がエラーなく文字列値を返し、`print-value` が `"おはよう、元気？"` を返す。
- `(evaluate "(list 'say \"アリス\" \"やあ\")")` が `(say アリス "やあ")` 相当を返し、結果走査で「シンボル `say` / シンボル `アリス`(または文字列) / 文字列 `"やあ"`」が区別できる。
- 長大文字列をループ生成する式が `max-cons` 超過で安全に停止する。

### R2. 最小の文字列操作（必須・最小セット）

著者がフラグや数値を本文へ動的に織り込めること（設計の「制御構造で会話を出し分け」を本文生成側でも少し支援）。

- **R2.1 `string-append`**: 複数文字列を連結（メモリ会計対象）。
- **R2.2 `number->string`**: 数値→文字列（カウンタや好感度の表示等）。
- （任意・あれば望ましい）`string-length`、`string=?`（R1.5 で代替可なら不要）。
- スコープ外（当面不要）: `substring`、`symbol<->string`、文字単位操作、正規表現等。必要になれば次増分で追加。

**受け入れ条件**: `(string-append "好感度: " (number->string 3))` が `"好感度: 3"` を返す。

### R3. 公開の値内省 API（必須）

ホストが評価結果（ocons ツリー）を走査してディレクティブへ変換するために、**`:wardlisp` パッケージから**結果の型判定と分解ができること。現状 ocons API は `wardlisp/src/types` にしか無く、文字列述語・取り出しも未提供。

- **R3.1**: `:wardlisp` から `ocons-p` / `ocons-ocar` / `ocons-ocdr`（または同等のリスト走査手段）を公開。
- **R3.2**: 文字列値の**述語**（例 `string-value-p`）と、**CL 文字列の取り出し**（例 `string-value`）を公開。
- **R3.3**: シンボル（=裸文字列）と数値を判定する手段（最低限、ホストが「これはタグ用シンボルか/文字列か/数値か」を分岐できること）。

**受け入れ条件**: ホスト(CL)が `evaluate` の返り値を、外部依存なく「リスト/文字列/数値/シンボル」に判別して再帰巡回できる（`wardlisp/src/types` への直接依存に頼らず `:wardlisp` 公開 API で完結）。

### R4. 評価時の初期束縛（あると良い・任意）

ホストが読者フラグを wardlisp 環境へ注入する手段。

- **R4.1**: `evaluate` が**初期束縛（alist 等）**を受け取り、評価前に環境へ束縛できる（例: `:bindings '(("met-alice" . t) ("affection" . 3))`）。
- 無い場合の代替: ホストが `(define met-alice t) …` を**文字列前置**して注入する（追加機能不要。ただし前置はパース/衝突リスクがありやや不格好）。

**受け入れ条件（R4 採用時）**: `:bindings` で渡した値がシーン式から変数参照できる。

### R5. セキュリティ / 非機能

- 文字列追加後も**既存のサンドボックス保証**（fuel/depth/cons/output/timeout）が維持される。文字列はメモリ予算に算入（R1.6）。出力 (`max-output`) は従来通り。
- 後方互換: 既存の数値/シンボル/リスト挙動・既存テスト・既存利用（SICP ノートブックの code/exercise セル）を壊さない。
- バージョニング: 破壊的変更を避け、追加で実現（`feat/string-type` 等のブランチ、PR を masatoi/wardlisp へ）。

## 非要件（wardlisp では作らない）

- ノベルのディレクティブ語彙（`say`/`bg`/`set-flag` 等）・インタプリタ・プレイヤー: **すべてホスト(recurya)側**の責務。wardlisp はあくまで「式を評価して汎用データを返す」だけ。
- フラグの永続化・読者状態: ホスト側（`novel_state`）。
- 副作用/IO/可変状態（`set-flag` 等）: wardlisp は純粋評価のまま。状態変化はホストがディレクティブとして解釈する。

## 受け入れ（このセッションの「要件決め」完了基準）

- ノベルエンジン設計が wardlisp に求める機能が R1〜R5 として列挙され、各々に受け入れ条件があること。
- 別セッションはこの要件から、wardlisp の**設計→プラン→実装（TDD）**を独立に進められること（必須: R1, R2, R3／任意: R4／非機能: R5）。
